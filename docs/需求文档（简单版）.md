# WeDownload 需求文档

> 一个自建的 HTTP 下载管理 Web 应用，部署在 N150 家庭服务器上。通过 Aria2 引擎实现远程 HTTP 直链下载，AriaNg 提供 Web 管理界面。
>
> 状态：阶段一已完成（Aria2 方案）  
> 最后更新：2026-07-19

---

## 一、背景与痛点

### 当前问题

| 痛点 | 描述 |
|------|------|
| 手机下载体验差 | 迅雷等 App 需要保持后台运行，耗电、发热严重 |
| 文件管理混乱 | 手机上下载的文件不好找，解压流程繁琐 |
| 加密压缩包 | 下载的压缩包需要密码解压，手机上操作困难 |
| 设备分离 | MacBook 是工作机，不想在上面下载 |
| 隐私问题 | 手机是私人物品，但 App 存储路径不直观 |

### 理想状态

- N150 服务器 7×24 挂机下载，不占用手机资源
- 手机上打开网页触发下载、查看进度、消费内容
- 一站式覆盖：添加任务 → 下载 → 解压（含密码）→ 浏览/播放
- 代码托管在独立 GitHub 仓库，部署流程：`git pull` → `systemctl restart`

## 二、使用场景

### 场景 1：下载影视并观看

1. 手机浏览器打开 WeDownload
2. 粘贴磁力链接，点击下载
3. 实时看到下载进度（速度、ETA、进度条）
4. 下载完成后点击文件 → HTML5 视频流播放
5. 看完可在页面内删除

### 场景 2：下载加密压缩包并浏览内容

1. 同上添加下载任务
2. 下载完成后，看到是一个 `.zip` / `.rar` / `.7z` 文件
3. 点击 → 弹出密码输入框
4. 输入密码 → 服务器解压 → 显示目录结构
5. 如果是漫画（图片序列）→ 进入翻页阅读器
6. 如果是视频 → 点击直接播放
7. 如果是文档 → 在线预览

### 场景 3：下载文档/书籍

1. 下载 PDF / EPUB
2. 点击 → 在线预览（PDF.js）或下载到手机本地

## 三、方案选择

### 3.1 整体策略：分阶段演进

| 阶段 | 内容 | 策略 |
|------|------|------|
| **阶段一（当前）** | 核心下载闭环 | 编排成熟开源组件，零代码 |
| **阶段二（后续）** | 自建 mobile-first 前端 + Node.js 后端 | 统一体验，按需开发 |

阶段一的目标：用最短路径跑通「手机远程控制 N150 下载 BT/磁力链接 → 取文件」的完整闭环，验证这个模式是否适合日常使用。阶段二再根据实际使用反馈决定自建前端的具体功能范围。

### 3.2 下载引擎：Aria2

**为什么从 qBittorrent（BT）切换到 Aria2（HTTP）？**

N150 在中国移动 CGNAT 后面，没有公网 IP。测试证实 BT 下载完全无法获取 peer（Ubuntu 官方 torrent 速度也是 0）。BT 协议要求至少一方有公网 IP，CGNAT 环境下几乎不可用。

Aria2 支持 HTTP/HTTPS 直链多线程下载，不受 CGNAT 影响。通过网盘离线下载（网盘服务器有公网 IP）获取 HTTP 直链，再丢给 N150 的 Aria2 下载，是当前网络环境下的可行方案。

| 特性 | 说明 |
|------|------|
| 引擎 | Aria2 1.37.0 |
| 协议 | HTTP / HTTPS / FTP / SFTP |
| RPC 接口 | JSON-RPC（端口 6800） |
| 多线程 | split=16, max-connection-per-server=16 |
| 断点续传 | 支持 |
| 资源占用 | ~5MB 内存 |

### 3.3 前端策略：先用现成的

| 组件 | 现成方案 | 说明 |
|------|---------|------|
| 下载管理 | **AriaNg**（Aria2 Web 前端） | 轻量，响应式，手机可用。通过自建 Python 代理同时提供静态文件 + RPC 转发 |

AriaNg 不是额外的服务，它只是 Aria2 的 Web 管理界面（一个 zip 包解压即可），通过 Python 代理与 Aria2 RPC 通信。

等阶段一跑通后，根据实际使用反馈决定是否自建前端。

### 3.4 网络穿透：Cloudflare Tunnel

**问题**：N150 在中国移动 NAT 后，没有公网 IP，80/443 端口被 ISP 封锁，无法直接对外提供 Web 服务。

**方案**：Cloudflare Tunnel（原名 Argo Tunnel）

```
你的手机 → https://wedownload.sherlockguo.com → Cloudflare CDN 边缘节点
                                                       ↑
                                             cloudflared 主动出站连接（HTTPS）
                                                       ↑
                                                  N150 服务器
```

**工作原理**：
- N150 上运行 `cloudflared` 程序，它**主动向外**连接 Cloudflare 的边缘节点
- 这是出站连接，对 ISP 来说就是普通的 HTTPS 流量，不受 80/443 封锁影响
- 域名 `sherlockguo.com` DNS 托管到 Cloudflare
- 配置 `wedownload.sherlockguo.com` → `http://localhost:8080`（qBittorrent Web UI）
- Cloudflare 自动签发 HTTPS 证书

**安全认证**：
- **qBittorrent 内置密码**：用户名/密码登录 Web UI
- Cloudflare Tunnel 隐藏 N150 真实 IP，外网无法直接扫描到服务器
- 单人使用场景，不需要额外的邮箱验证或 Geo-block

## 五、技术选型

### 阶段一（当前）

| 层 | 技术 | 理由 |
|----|------|------|
| 下载引擎 | Aria2 | HTTP/HTTPS 多线程下载，JSON-RPC，不受 CGNAT 影响 |
| 下载管理 UI | AriaNg | Aria2 的 Web 管理界面，响应式，手机可用 |
| RPC 代理 | Python http.server | 静态文件 + JSON-RPC 转发到 Aria2（端口 6800） |
| 内网穿透 | Cloudflare Tunnel（cloudflared） | 无需公网 IP/VPS |
| 进程管理 | systemd | aria2 + ariang 两个服务 |
| DNS | Cloudflare | 域名托管 + 自动 HTTPS 证书 |
| CI/CD | GitHub Actions | push → Release → webhook + cron |
| 监控 | WeMonitor | 部署状态 + TCP 健康检查（端口 8080） |

### 阶段二（后续，暂不实施）

| 层 | 技术 | 理由 |
|----|------|------|
| 后端 | Node.js + Express | 和现有 WeMusic 项目技术栈一致 |
| 前端 | 纯 HTML/CSS/JS（无框架） | 轻量，mobile-first |
| 实时通信 | WebSocket（ws 库） | 推送下载进度 |
| 解压 | 7z（7-Zip CLI） | 支持 zip/rar/7z + 密码 |
| 视频播放 | HTML5 Video + Range 请求 | 原生支持 |
| PDF 预览 | PDF.js | 成熟稳定 |

## 七、非功能需求

- **安全**：Cloudflare Tunnel 隐藏 N150 真实 IP + qBittorrent 内置密码认证
- **性能**：qBittorrent 内存 ~150MB，cloudflared ~30MB，合计 ~180MB，在 7.5GB 内存下无压力
- **存储**：~202GB 可用空间作为下载目录，后续可按需扩容
- **维护**：systemd 管理两个服务（qBittorrent-nox + cloudflared），开机自启，异常自动重启
- **部署**：GitHub Actions CI 自动打包发布（commit SHA tag），webhook 通知 N150 → `deploy-agent.sh` 自动下载部署（ghproxy 镜像 + 直连 fallback），cron 兜底
- **监控**：已纳入 WeMonitor 管理（部署状态 + TCP 健康检查 8080 端口）
