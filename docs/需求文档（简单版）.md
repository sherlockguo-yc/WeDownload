# WeDownload 需求文档

> 一个自建的移动端优先的 BT/磁力下载管理 Web 应用，部署在 N150 家庭服务器上。
>
> 状态：需求讨论阶段  
> 最后更新：2026-07-19

---

## 一、背景与痛点

### 当前问题

| 痛点 | 描述 |
|------|------|
| 手机下载体验差 | 迅雷等 App 需要保持后台运行，耗电、发热严重 |
| 文件管理混乱 | 手机上下载的文件不好找，解压流程繁琐 |
| 加密压缩包 | 下载的压缩包需要密码解压，手机上操作困难 |
| 设备分离 | MacBook 是工作机，不想在上面下载 |
| 隐私问题 | 手机是私人物品，但 App 存储路径不直观 |

### 理想状态

- N150 服务器 7×24 挂机下载，不占用手机资源
- 手机上打开网页触发下载、查看进度、消费内容
- 一站式覆盖：添加任务 → 下载 → 解压（含密码）→ 浏览/播放
- 代码托管在独立 GitHub 仓库，部署流程：`git pull` → `systemctl restart`

## 二、使用场景

### 场景 1：下载影视并观看

1. 手机浏览器打开 WeDownload
2. 粘贴磁力链接，点击下载
3. 实时看到下载进度（速度、ETA、进度条）
4. 下载完成后点击文件 → HTML5 视频流播放
5. 看完可在页面内删除

### 场景 2：下载加密压缩包并浏览内容

1. 同上添加下载任务
2. 下载完成后，看到是一个 `.zip` / `.rar` / `.7z` 文件
3. 点击 → 弹出密码输入框
4. 输入密码 → 服务器解压 → 显示目录结构
5. 如果是漫画（图片序列）→ 进入翻页阅读器
6. 如果是视频 → 点击直接播放
7. 如果是文档 → 在线预览

### 场景 3：下载文档/书籍

1. 下载 PDF / EPUB
2. 点击 → 在线预览（PDF.js）或下载到手机本地

## 三、方案选择

### 3.1 整体策略：分阶段演进

| 阶段 | 内容 | 策略 |
|------|------|------|
| **阶段一（当前）** | 核心下载闭环 | 编排成熟开源组件，零代码 |
| **阶段二（后续）** | 自建 mobile-first 前端 + Node.js 后端 | 统一体验，按需开发 |

阶段一的目标：用最短路径跑通「手机远程控制 N150 下载 BT/磁力链接 → 取文件」的完整闭环，验证这个模式是否适合日常使用。阶段二再根据实际使用反馈决定自建前端的具体功能范围。

### 3.2 下载引擎：qBittorrent-nox

**qBittorrent-nox 是什么？** qBittorrent 是一个开源 BT 下载客户端（类似迅雷），**-nox 版本去掉了桌面 GUI，仅保留下载引擎 + Web 管理界面**，专为 NAS/服务器场景设计。它跑在 N150 上，下载的文件直接存在 N150 的本地磁盘（如 `~/downloads/`）。

下载引擎选型：

| 维度 | Aria2 | **qBittorrent-nox（选择）** |
|------|-------|---------------------------|
| BT 引擎 | 自研实现，更新缓慢 | **libtorrent**（业界最活跃的 BT 库，15 年积累） |
| DHT / PEX / LSD | 基础支持 | **更完善**，去中心化节点发现更强 |
| 协议加密 | 支持 | **更成熟**，对 ISP 的 DPI 限速规避更好 |
| 稳定性 | 偶尔卡在 99% | **Endgame mode 防卡死**，Recheck 校验完整性 |
| 多协议 | ✅ HTTP/FTP/BT/Magnet | ❌ 仅 BT/Magnet |
| 远程控制接口 | JSON-RPC | Web API（REST） |
| 用户认证 | 无（需额外加层） | **内置用户名/密码** |
| 资源占用 | ~50MB | ~150MB（N150 无压力） |

**选择理由**：当前阶段只需 BT/磁力下载，HTTP 直链需求后面单独加 Aria2。qBittorrent 的 libtorrent 引擎在 BT 协议栈成熟度上明显更强，直接对应「稳定性 > 峰值速度」的核心诉求。

### 3.3 前端策略：先用现成的

| 组件 | 现成方案 | 说明 |
|------|---------|------|
| 下载管理 + 文件浏览 | **VueTorrent**（qBittorrent 的第三方 Web 皮肤） | 现代风格，响应式，手机可用。自带任务管理和文件列表（右键下载） |

**VueTorrent 不是额外的服务**，它只是 qBittorrent Web UI 的一个替代皮肤（一个 zip 包解压到配置目录即可），不增加维护负担。

等阶段一跑通后，根据实际使用反馈决定是否自建前端。如果 VueTorrent 在手机上够用，就不重复造轮子；如果体验有明显短板，再针对痛点自建。

### 3.4 网络穿透：Cloudflare Tunnel

**问题**：N150 在中国移动 NAT 后，没有公网 IP，80/443 端口被 ISP 封锁，无法直接对外提供 Web 服务。

**方案**：Cloudflare Tunnel（原名 Argo Tunnel）

```
你的手机 → https://wedownload.sherlockguo.com → Cloudflare CDN 边缘节点
                                                       ↑
                                             cloudflared 主动出站连接（HTTPS）
                                                       ↑
                                                  N150 服务器
```

**工作原理**：
- N150 上运行 `cloudflared` 程序，它**主动向外**连接 Cloudflare 的边缘节点
- 这是出站连接，对 ISP 来说就是普通的 HTTPS 流量，不受 80/443 封锁影响
- 域名 `sherlockguo.com` DNS 托管到 Cloudflare
- 配置 `wedownload.sherlockguo.com` → `http://localhost:8080`（qBittorrent Web UI）
- Cloudflare 自动签发 HTTPS 证书

**安全认证**：
- **qBittorrent 内置密码**：用户名/密码登录 Web UI
- Cloudflare Tunnel 隐藏 N150 真实 IP，外网无法直接扫描到服务器
- 单人使用场景，不需要额外的邮箱验证或 Geo-block

### 3.5 架构概要（阶段一）

```
┌──────────────────────────────────────────────────────────────┐
│                    任何有浏览器的设备                          │
│               https://wedownload.sherlockguo.com              │
└──────────────────────────┬───────────────────────────────────┘
                           │ HTTPS
                           ▼
┌──────────────────────────────────────────────────────────────┐
│                      Cloudflare CDN                           │
│          Cloudflare Access（邮箱验证码，第一层认证）            │
└──────────────────────────┬───────────────────────────────────┘
                           │ cloudflared 出站隧道
                           ▼
┌──────────────────────────────────────────────────────────────┐
│                       N150 家庭服务器                          │
│                                                              │
│  ┌─────────────────────┐                                     │
│  │  qBittorrent-nox     │                                     │
│  │  + VueTorrent 皮肤   │                                     │
│  │  （BT/磁力下载引擎    │                                     │
│  │   + 任务管理          │                                     │
│  │   + 文件浏览/下载）   │                                     │
│  │  Web UI: 8080        │                                     │
│  └──────────┬──────────┘                                     │
│             │ 读写                                            │
│             ▼                                                │
│  ┌────────────────────────────────────┐                      │
│  │           ~/downloads/             │                      │
│  │         （N150 本地磁盘目录）       │                      │
│  └────────────────────────────────────┘                      │
│                                                              │
│  ┌────────────────────────────────────┐                      │
│  │           cloudflared               │                      │
│  │  Tunnel: wedownload → localhost:8080│                      │
│  └────────────────────────────────────┘                      │
└──────────────────────────────────────────────────────────────┘
```

**阶段一只需安装两个程序**（qBittorrent-nox + cloudflared），通过 systemd 管理。不需要写任何代码。Node.js 后端和自建前端留到阶段二。

## 四、功能模块

### 阶段一：核心下载闭环（零代码，现成组件）

|------|---------|

| 功能 | 描述 |

| 深色模式 | 支持浅色/深色切换 |

## 五、技术选型

### 阶段一（当前）

| 层 | 技术 | 理由 |
|----|------|------|
| 下载引擎 | qBittorrent-nox（libtorrent） | 业界最成熟的 BT 协议实现，稳定性优先 |
| 下载管理 + 文件浏览 | VueTorrent（qBittorrent Web UI 皮肤） | 现代风格，响应式，手机可用。自带任务管理 + 文件列表（右键下载） |
| 内网穿透 | Cloudflare Tunnel（cloudflared） | 无需公网 IP/VPS，出站连接绕过 ISP 端口封锁 |
| 安全认证 | qBittorrent 内置密码 | 单人使用场景，无需额外认证层 |
| 进程管理 | systemd | 管理 qBittorrent-nox + cloudflared 两个服务的启停和开机自启 |
| DNS | Cloudflare | 域名托管 + 自动 HTTPS 证书 |
| CI/CD | GitHub Actions | push 自动打包 → 发布 Release → N150 cron 自动拉取更新（和 WeMusic/WeMonitor 一致） |

### 阶段二（后续，暂不实施）

| 层 | 技术 | 理由 |
|----|------|------|
| 后端 | Node.js + Express | 和现有 WeMusic 项目技术栈一致 |
| 前端 | 纯 HTML/CSS/JS（无框架） | 轻量，mobile-first |
| 实时通信 | WebSocket（ws 库） | 推送下载进度 |
| 解压 | 7z（7-Zip CLI） | 支持 zip/rar/7z + 密码 |
| 视频播放 | HTML5 Video + Range 请求 | 原生支持 |
| PDF 预览 | PDF.js | 成熟稳定 |
| HTTP 直链下载 | Aria2 | 作为补充引擎，仅处理 HTTP/FTP |

## 七、非功能需求

- **安全**：Cloudflare Tunnel 隐藏 N150 真实 IP + qBittorrent 内置密码认证
- **性能**：qBittorrent 内存 ~150MB，cloudflared ~30MB，合计 ~180MB，在 7.5GB 内存下无压力
- **存储**：~202GB 可用空间作为下载目录，后续可按需扩容
- **维护**：systemd 管理两个服务（qBittorrent-nox + cloudflared），开机自启，异常自动重启
- **部署**：GitHub Actions CI 自动打包发布，N150 cron 每分钟自动检查更新，和 WeMusic/WeMonitor 完全一致的流程

## 八、实施计划（阶段一）

### 第一步：安装服务 ✅ 已完成

1. **qBittorrent-nox**：`sudo apt install qbittorrent-nox`（Ubuntu 24.04 默认源包含）
2. 安装 VueTorrent 皮肤：下载 zip → 解压到 `~/.config/qBittorrent/vuetorrent/`
3. UFW 放行 8080 端口：`sudo ufw allow 8080/tcp`
4. 局域网验证 qBittorrent Web UI（`http://192.168.31.102:8080`）可访问

### 第二步：systemd 管理 ✅ 已完成

5. 为 qBittorrent-nox 写 systemd unit 文件（`/etc/systemd/system/qbittorrent-nox.service`）
6. `systemctl enable --now qbittorrent-nox`
7. 重启 N150 验证开机自启正常

### 第三步：网络穿透 ✅ 已完成

8. cloudflared 已安装（v2026.7.1）
9. 复用现有 Tunnel（与 WeMusic/WeMonitor 共用，tunnel ID: `8d17217c-96d0-4433-ac02-540f8e539f1c`）
10. 添加 `wedownload.sherlockguo.com` 路由到 `/etc/cloudflared/config.yml`
11. 修复 systemd 启动超时问题：
    - `TimeoutStartSec` 从 15 改为 60
    - 协议改为 `--protocol http2`（QUIC 跨地区连接不稳定）
12. 在 Cloudflare DNS 中添加 CNAME：`wedownload.sherlockguo.com` → tunnel
13. 验证 `https://wedownload.sherlockguo.com` 返回 200

### 第四步：安全认证 ⬜ 已确认跳过

14. 跳过 Cloudflare Access（Zero Trust Free 计划不支持 One-time PIN；单人使用场景不需要额外邮箱验证）
15. 跳过 Geo-block（防止香港等海外节点访问被误拦）
16. 当前安全策略：Cloudflare Tunnel 隐藏真实 IP + qBittorrent 密码认证，满足单人使用需求

### 第五步：CI/CD 自动化（待实施，约 20 分钟）

### 第六步：调优（按需迭代）
